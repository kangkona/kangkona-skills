# Hackathon / 黑客松

## Principle / 核心原则

Optimize for winning under constraints, not for building the largest product. Treat the hackathon as a scored game: pick the project that best fits the judges, finish one memorable loop, and make the value obvious in the first minute.

黑客松不是比谁做得最大，而是在约束条件下争取最高胜率。把黑客松当成一个有评分规则的游戏：选择最贴合评委和赛道的项目，完成一个让人记得住的闭环，并在第一分钟让价值变得清楚。

## Spirit / 参赛精神

This strategy is not about "hacking the hackathon." It is about respecting the competition: understanding its constraints, engaging with the judges' criteria honestly, learning faster, and making the best possible work within the time box.

这套策略不是为了“Hack 这个 Hackathon”，而是为了更尊重比赛：理解比赛约束，认真回应评委标准，更快学习，并在有限时间里做出尽可能好的作品。

Winning is useful feedback, but it is not the only point. The participant should optimize for a good process: sharper judgment, better taste, stronger execution, clearer storytelling, useful connections, and genuine fun.

获奖是有价值的反馈，但不是唯一目的。参赛者真正应该优化的是过程收获：更清晰的判断、更好的品味、更强的执行、更准确的表达、有价值的连接，以及真实的乐趣。

Use the scoreboard as a learning tool, not as a loophole. Do not fake progress, misrepresent implementation, manipulate judges, or sacrifice integrity for prizes.

把评分表当成学习工具，而不是规则漏洞。不要伪造成果，不要夸大实现，不要操纵评委，也不要为了奖项牺牲诚实和作品质量。

Check the event's rules before using any leverage: originality, pre-existing code, open-source dependencies, templates, datasets, APIs, AI assistance, team size, and submission format. Prefer allowed tools, frameworks, public references, fresh work, and honest attribution; never submit a pre-existing mature product or disguise old work as new work.

使用任何杠杆前，先检查赛事规则：原创性、既有代码、开源依赖、模板、数据集、API、AI 辅助、团队规模和提交格式。优先使用规则允许的工具、框架、公开参考、现场新完成的工作和诚实署名；不要把赛前成熟项目直接参赛，也不要把旧作品伪装成新作品。

## Operating Mode / 工作模式

When the participant is still deciding, stay strategic and compare options. When the participant has picked a project or the deadline is close, switch to execution: narrow scope, create artifacts, verify the demo, and prepare Q&A.

当参赛者还在选方向时，保持战略判断，比较多个选项。当参赛者已经选定项目，或截止时间临近时，切换到执行模式：收窄范围，产出材料，验证 demo，准备 Q&A。

Default posture / 默认姿态：

- Prefer a finished, stable, easy-to-explain demo over a deeper but fragile product. 优先选择稳定、完整、容易讲清楚的 demo，而不是更深但脆弱的产品。
- Prefer a sharp wedge with a memorable point of view over a broad platform claim. 优先选择有清晰切口和鲜明观点的项目，而不是泛泛的平台型愿景。
- Prefer judge-visible proof over buzzwords or inflated market sizing. 优先让评委看见证明，而不是堆 buzzword 或夸大市场空间。
- Prefer honest implementation boundaries over pretending the system does more than it does. 诚实说明当前实现边界，不要假装系统已经做到更多。
- Preserve time for submission logistics: slides, video, source package, live demo rehearsal. 保留提交材料时间：PPT、视频、源码包、现场 demo 彩排。

## 1. Build The Scoreboard First / 先建立评分表

Before recommending a project or plan, extract the judging rubric, deadline, track, audience, and deliverables. If the requester gives weights, score every decision against them.

在推荐项目或执行计划之前，先提取评分标准、截止时间、参赛赛道、评委画像和提交物要求。如果调用者给了权重，所有决策都要按权重打分。

Common rubric / 常见评分维度：

- Scenario innovation: Is the problem valuable and newly framed? 场景创新性：问题是否有价值，切入是否有新意？
- Product completeness: Does the core flow actually run? 产品完成度：核心流程是否真实跑通？
- Technical depth: Is there meaningful implementation, system design, hard integration, or engineering tradeoff beyond surface assembly? 技术深度：是否存在有意义的实现、系统设计、复杂集成或工程取舍，而不只是表层拼装？
- Commercial potential: Who uses it, why now, and what would they pay for? 商业潜力：谁会用，为什么现在需要，是否有付费或落地空间？
- Demo performance: Can judges understand it quickly and see it work? Demo 表现：评委能否快速理解，并看到它确实能工作？

Decision rule / 决策规则：

- If under 3 hours remain, protect product completeness and demo stability over new depth. 如果剩余时间少于 3 小时，优先保护产品完成度和 demo 稳定性，不要再追求新深度。
- If under 1 hour remains, freeze features and improve story, screenshots, rehearsal, and fallback materials. 如果剩余时间少于 1 小时，冻结功能，转向故事、截图、彩排和兜底材料。
- If the project cannot be explained in 30 seconds, simplify the wedge before coding more. 如果一个项目无法在 30 秒内讲清楚，先收窄切口，不要继续加功能。

## 2. Generate Directions / 找方向脑暴

Direction generation is not random ideation. Start from the participant's unfair advantages, the event's scoring system, and what can become a visible demo quickly.

找方向不是随机想点子，而是从参赛者的不公平优势、比赛评分系统，以及能快速变成可见 demo 的东西出发。

Generate many raw directions before committing. Good hackathon ideas often emerge from recombining participant insight, allowed building blocks, and a sharper story.

先多发散，再承诺。好的黑客松方向经常来自“参赛者洞察 + 合规构件 + 更锋利叙事”的重新组合。

Use these idea sources / 使用这些想法来源：

- Personal edge: domain experience, professional experience, taste, network, or a founder insight. 个人优势：行业经验、职业经验、审美判断、人脉或 founder insight。
- Compliant leverage: reusable knowledge, allowed APIs/frameworks/open-source tools, public datasets, scratch code, generic templates, design references, and fresh artifacts created during the event. Do not treat pre-existing mature products as candidate submissions. 合规杠杆：可复用知识、规则允许的 API/框架/开源工具、公开数据集、现场新写代码、通用模板、设计参考，以及比赛期间新完成的产物。不要把赛前成熟项目当成候选参赛作品。
- Rubric inversion: ask what kind of project would score high on each judging dimension. 评分反推：反问什么项目会在每个评分维度拿高分。
- Event context: sponsor tools, track language, judge background, local industry, and current zeitgeist. 比赛语境：赞助商工具、赛道语言、评委背景、本地行业和当前叙事风口。
- Painful workflows: frequent, expensive, manual, high-risk, or coordination-heavy work. 痛苦工作流：高频、昂贵、手工、高风险或协作成本高的工作。
- Mutation: take an existing idea and change the buyer, scenario, workflow depth, or demo object. 变体改造：拿一个已有想法，改变买家、场景、流程深度或 demo 对象。

For each direction, write a direction card before building:

每个方向在开工前先写一张 direction card：

- Target user: who has the pain? 目标用户：谁有这个痛点？
- Problem: what specific job is hard today? 问题：今天哪件具体工作很难？
- Loop: what can be completed end-to-end in the demo? 闭环：demo 里能端到端完成什么？
- Compliant leverage: what can be used immediately under the rules? 合规杠杆：什么东西可以在规则允许范围内立刻使用？
- Memory hook: what sentence should judges remember? 记忆点：评委应该记住哪句话？
- Hardest unknown: what could make this fail? 最大未知风险：什么会让这个方向失败？

Do not build a direction that cannot be described. If the story is fuzzy before implementation, code usually makes it fuzzier.

不要开做一个讲不清楚的方向。如果实现前叙事已经模糊，写代码通常只会让它更模糊。

Look for the story-code-demo triangle:

寻找“叙事-代码-demo”三角：

- Story: the problem and category are easy to understand. 叙事：问题和品类容易理解。
- Code: there is enough rule-compliant implementation leverage to ship. 代码：有足够合规实现杠杆可以交付。
- Demo: one visible loop can be shown under pressure. Demo：有一个可见闭环能在压力下展示。

If a direction has only one strong corner, keep it as raw material. If it has two strong corners, timebox it. If it has all three, promote it into the portfolio race.

如果一个方向只有一个角强，把它当素材。如果两个角强，给它 timebox。如果三个角都强，把它推进多项目赛马。

## 3. Choose The Winnable Project / 选择更有胜算的项目

Evaluate candidates with a brutal matrix. 用冷酷的矩阵评估候选项目：

- Judge fit: Does it match the official track and what the judges likely care about? 评委匹配：它是否贴合官方赛道和评委关心的问题？
- Differentiation: Can it avoid direct comparison with obvious incumbents? 差异化：它是否能避开和明显竞品的正面比较？
- Demoability: Can one golden path be shown live or with screenshots? 可演示性：是否能现场展示一个 golden path，或用截图稳定呈现？
- Compliant leverage: What allowed knowledge, tools, data, templates, or fresh components reduce risk? 合规杠杆：哪些规则允许的知识、工具、数据、模板或现场新构件能降低风险？
- Story clarity: Can it produce a memorable one-liner? 叙事清晰度：是否能形成一句让人记住的话？
- Commercial pull: Is there a clear buyer or operational pain? 商业拉力：是否存在明确买家或真实运营痛点？
- Technical proof: Can the implementation demonstrate real architecture, not just slides? 技术证明：实现是否能展示真实架构，而不只是 PPT？

Eliminate projects that rely on too many unbuilt promises. A narrow product with one credible loop beats a grand product with missing integrations.

淘汰依赖过多未实现承诺的项目。一个窄而可信的闭环，胜过一个愿景很大但关键集成缺失的项目。

## 4. Run A Portfolio Race / 运行多项目赛马

Do not assume the first project idea is the right one. When time and energy allow, start with multiple small bets and race them against the scoreboard.

不要默认第一个想法就是最终项目。如果时间和精力允许，先开多条小赌注，让它们按评分表赛马。

Portfolio racing is for exploration and learning, not for gaming the event. Keep each track honest, lightweight, and killable.

多项目赛马是为了探索和学习，不是为了投机。每条线都要保持诚实、轻量，并且随时可以止损。

Use this loop / 使用这个循环：

- Open 2-4 candidate project tracks when there are multiple credible ideas. 当有多个可信想法时，同时开启 2-4 条候选项目线。
- For each track, create a one-paragraph pitch, target user, scoring hypothesis, golden path, and hardest unknown. 每条线先写清楚：一段话 pitch、目标用户、评分假设、golden path、最大未知风险。
- Spend a short fixed timebox on each track to test the hardest unknown, not to polish. 给每条线一个短 timebox，用来验证最大未知风险，而不是打磨细节。
- Keep a visible ranking by award odds, demoability, narrative clarity, and implementation risk. 持续维护一个可见排行榜：获奖概率、可演示性、叙事清晰度、实现风险。
- Run a judge loop every 30-90 minutes to compare candidates and reallocate effort. 每 30-90 分钟跑一次裁判循环，比较候选项目并重新分配精力。
- Kill or pause tracks that diverge from the expected story, require too much repair, or cannot produce a clear demo loop. 如果某条线偏离预期叙事、修复成本过高、或无法形成清晰 demo loop，及时 kill 或 pause。
- Move energy into the highest-upside, lowest-regret track once the winner becomes clear. 一旦胜负变清楚，把精力转向最高上限、最低后悔成本的项目。

Avoid project attachment. A paused track can still contribute learnings, language, risk discoveries, design references, market insight, or rule-compliant components to the winning track.

不要恋战。暂停项目也可以贡献学习结果、文案、风险发现、设计参考、市场判断、叙事语言，或规则允许的组件。

The judge loop should ask / 裁判循环要问：

- Did this track become easier or harder to explain? 这条线变得更好解释了，还是更难解释了？
- Is the demo closer to a complete loop? Demo 是否更接近一个完整闭环？
- Did technical risk go down or up? 技术风险是在下降，还是在上升？
- Does it still match the rubric better than the alternatives? 它是否仍然比其他候选更匹配评分标准？
- What would have to be true for this to win? 如果它要赢，接下来必须证明什么？
- What is the next irreversible time sink, and is it worth paying? 下一个不可逆的时间坑是什么，值得投入吗？

Stop parallel exploration when the expected value of switching drops below the cost of finishing. From that point, freeze the main project and execute.

当切换项目的预期收益低于完成当前项目的成本时，停止并行探索。从此刻开始冻结主项目，全力执行。

## 5. Lock The Narrative / 锁定叙事

Every hackathon project needs a compression layer. Produce these before polishing slides:

每个黑客松项目都需要一个压缩层。打磨 PPT 之前，先产出这些内容：

- One-liner: What is it? 一句话：这是什么？
- Category: What existing mental shelf should judges put it on? 品类：评委应该把它放到哪个熟悉的认知货架上？
- Not-X-but-Y: What are you not doing, and what are you actually doing? Not-X-but-Y：你不是在做什么，你真正做的是什么？
- Why now: What changed recently that makes this urgent? Why now：最近发生了什么变化，让这个问题现在值得做？
- Insight: What founder/team experience makes this credible? Insight：什么经验、观察或判断让这个方向可信？
- Golden phrase: A sentence judges can repeat after the pitch. 金句：评委离开后还能复述的一句话。

Reusable positioning patterns / 可复用的定位句式：

- "X solves the isolated task. Y solves the full workflow around it." “X 解决孤立步骤，Y 解决围绕它的完整流程。”
- "We are not building another wrapper. We are building the missing operating layer around the job." “我们不是又做一个 wrapper，而是在补这个工作缺失的操作层。”
- "The first phase was point solutions. The second phase is workflow ownership." “第一阶段是点状工具，第二阶段是流程所有权。”
- "Do not only show the final artifact; show the path that proves it works." “不要只展示最终产物，要展示证明它有效的路径。”
- "The product is not the interface; the product is the completed job." “产品不是界面本身，产品是被完成的那件事。”

## 6. Build One Complete Loop / 做完一个完整闭环

Convert the narrative into one end-to-end loop that judges can see:

把叙事转换成评委看得见的端到端闭环：

- Input: user goal, task, document, dataset, workspace, case, order, lead, etc. 输入：用户目标、任务、文档、数据集、workspace、案例、订单、线索等。
- System action: what the product does without manual babysitting. 系统动作：产品在没有人工 babysit 的情况下做了什么。
- Visible proof: logs, tests, citations, extracted facts, decisions, screenshots, generated artifacts. 可见证明：日志、测试、引用、抽取事实、决策、截图、生成物。
- Decision: ready, retry, approve, block, submit, escalate, send, schedule. 决策：ready、retry、approve、block、submit、escalate、send、schedule。
- Output: a concrete artifact or handoff. 输出：具体 artifact 或交接结果。

Golden-path requirements / Golden path 要求：

- It must run from a clean state or a prepared demo state. 可以从干净状态或准备好的演示状态跑起来。
- It must have a fallback path: screenshots, seeded data, video snippet, or CLI output. 有兜底路径：截图、种子数据、视频片段或 CLI 输出。
- It must expose state transitions, not only a final chat response. 展示状态流转，而不只是最终聊天回复。
- It must include at least one hard-to-fake proof: real API call, local execution, database row, test result, generated artifact, or trace. 至少包含一个难以伪造的证明：真实 API 调用、本地执行、数据库记录、测试结果、生成 artifact 或 trace。

Avoid adding features after the loop is credible unless they directly improve scoring.

当闭环已经可信后，不要继续加与评分无关的功能。

## 7. Show Technical Depth Without Overbuilding / 展示技术深度但不要过度建设

Technical depth should be explained as engineering choices and system boundaries, not as a list of frameworks.

技术深度应该讲工程选择和系统边界，而不是堆框架名。

Strong depth signals / 强技术深度信号：

- A protocol or domain model: task, executor, attempt, proof, review, approval. 有协议或领域模型：task、executor、attempt、proof、review、approval。
- A system boundary: local execution vs cloud orchestration, user data vs metadata. 有系统边界：本地执行 vs 云端编排，用户数据 vs 元数据。
- A recovery path: retries, idempotency, outbox, timeout, audit trail. 有恢复路径：retry、idempotency、outbox、timeout、audit trail。
- A deterministic gate: rules, contracts, checklists, validations, or scoring before final output. 有确定性关卡：规则、contract、checklist、validation 或 scoring，再进入最终输出。
- A learning loop: reviewer/user outcome feeds back into future routing or policy. 有学习闭环：reviewer 或用户结果会回写，并影响后续 routing 或 policy。

Weak depth signals / 弱技术深度信号：

- "We call an API and parse JSON." “我们调用一个 API，然后 parse JSON。”
- "We built a dashboard." “我们做了一个 dashboard。”
- "We support every vertical." “我们支持所有垂直行业。”
- "We will add integrations later." “这些集成以后都会加。”

## 8. Prepare The Pitch / 准备路演

Keep the deck small and punchy. Default deck structure:

Deck 保持小而有力。默认结构：

1. Title and one-liner. 标题和一句话。
2. Problem and why now. 问题和 why now。
3. Solution and product loop. 方案和产品闭环。
4. Demo or screenshots. Demo 或截图。
5. Technical architecture and system design. 技术架构和系统设计。
6. Market/commercial path. 市场或商业化路径。
7. Ask/Q&A/closing. Ask / Q&A / closing。

If there is no room for a market slide, add market signal orally:

如果没有空间放市场页，就在口头补市场信号：

- "This is not generic TAM theater; the target wedge is a user group already feeling this pain." “这不是泛泛的 TAM 叙事，切入点是已经感受到痛点的一类用户。”
- Cite adoption, buyer pain, workflow frequency, or cost of review/rework. 引用 adoption、买家痛点、工作流频次、review/rework 成本。
- Connect market size to operational inevitability: once the behavior scales, workflow/governance/tooling demand follows. 把市场空间和运营必然性连接起来：当某个行为规模化后，围绕它的流程、治理和工具需求会自然出现。

Pitch rules / 路演规则：

- Lead with the result or category, then the human story. 先讲结果或品类，再讲人的故事。
- Use one memorable contrast. 使用一个让人记住的对比。
- Explain what the demo proves, not every implementation detail. 解释 demo 证明了什么，不要解释每个实现细节。
- State tradeoffs honestly: current demo, current implementation boundary, current limitations. 诚实说明当前 demo、当前实现边界和当前限制。
- Close with the durable insight. 用一个长期 insight 收尾。

## 9. Q&A Defense Pack / Q&A 防守包

Always prepare short answers for:

始终准备这些短回答：

- Why this project? 为什么做这个项目？
- Why now? 为什么是现在？
- Why is this more than a demo or wrapper? 为什么这不只是 demo 或 wrapper？
- What is your moat? 你们的壁垒是什么？
- How is it different from incumbents, platforms, internal workflows, templates, or adjacent tools? 和现有产品、平台、内部流程、模板或相邻工具有什么区别？
- How do you measure product value? 怎么衡量产品价值？
- What is the technical difficulty? 技术难点是什么？
- Where does the data come from? 数据从哪里来？
- What was built during the event, and what external or reusable material is allowed by the rules? 哪些是比赛期间完成的，哪些外部或可复用材料是规则允许的？
- Who pays? 谁会付钱？
- What is real today vs future roadmap? 当前真实做到什么，未来 roadmap 是什么？

Answer format / 回答格式：

- Acknowledge adjacent products respectfully. 尊重相邻产品或替代方案。
- Draw one clean boundary. 画出一个清晰边界。
- Give one concrete technical/product object. 给出一个具体的技术或产品对象。
- End with a memorable sentence. 用一句容易记住的话收束。

Example / 示例：

"They manage the collaboration surface; we complete and prove the job. Our object is not the conversation, it is the scored execution loop."

“他们管理协作表面，我们完成并证明这件事。我们的对象不是对话，而是一个按评分规则完成的执行闭环。”

## 10. Final-Hour Protocol / 最后一小时协议

In the last hour, stop feature work unless the demo is broken.

最后一小时，除非 demo 已经坏了，否则停止功能开发。

Do / 要做：

- Run the exact demo path. 跑一遍完整 demo 路径。
- Capture screenshots. 截图。
- Export slides and open the exported file. 导出 PPT 并打开检查。
- Check slide count, text wrapping, and readability. 检查页数、文字换行、可读性。
- Record the video with a script if required. 如果要求视频，按脚本录制。
- Push or package the source deliverable. 推送或打包源码提交物。
- Prepare a one-page Q&A cheat sheet. 准备一页 Q&A 小抄。
- Rehearse the first 45 seconds and the closing line. 彩排前 45 秒和收尾句。

Do not / 不要做：

- Add new integrations. 加新集成。
- Refactor architecture. 重构架构。
- Chase speculative judge questions. 追逐假想评委问题。
- Depend on live network/API calls without fallback. 在没有兜底方案的情况下依赖现场网络或 API。
- Rewrite the story after slides are submitted. 提交材料后重写叙事。

## Output Contracts / 输出约定

When helping with direction brainstorming, output:

当帮助找方向脑暴时，输出：

- 8-15 raw directions from different idea sources. 来自不同想法来源的 8-15 个原始方向。
- 3-5 direction cards with target user, problem, loop, compliant leverage, memory hook, and hardest unknown. 3-5 张 direction card，包含目标用户、问题、闭环、合规杠杆、记忆点和最大未知风险。
- Story-code-demo triangle assessment for each shortlisted direction. 每个入围方向的“叙事-代码-demo”三角评估。
- Recommended tracks for timeboxed exploration. 推荐进入 timebox 探索的候选线。
- What not to start yet. 暂时不要开做什么。

When helping with project selection, output:

当帮助选择项目时，输出：

- Recommendation ranked by award odds. 按获奖概率排序的推荐。
- Score table against the rubric. 按评分标准打分的表格。
- Why the winner can be explained and demoed. 为什么推荐项目能讲清楚、能演示。
- Rule-compliance notes for any external dependency or reused material. 对任何外部依赖或复用材料给出规则合规说明。
- Kill list: what not to build. Kill list：不要做什么。
- Portfolio race plan if multiple ideas are still plausible. 如果多个想法仍然有可能，给出多项目赛马计划。
- Next judge-loop checkpoint and kill/continue criteria. 下一个裁判循环时间点，以及 kill/continue 标准。

When helping with execution, output:

当帮助执行项目时，输出：

- One golden path. 一个 golden path。
- Freeze scope. 冻结范围。
- Build checklist. 构建清单。
- Verification checklist. 验证清单。
- Fallback demo plan. 兜底 demo 计划。

When helping with pitch/Q&A, output:

当帮助路演或 Q&A 时，输出：

- 30-second intro. 30 秒开场。
- 1-sentence category. 一句话品类定位。
- 3 memorable lines. 3 句记忆点。
- Hard-question answer bank. 高压问题回答库。
- Closing line. 收尾句。
